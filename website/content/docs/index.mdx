깨끗한 클론에서 5분 안에 첫 `secret run`까지 도달합니다. 서버 하나가 API와 대시보드(`/`)를 함께 서빙하므로 별도 서비스를 배포할 필요가 없습니다.

<Callout type="info">
대상은 단일 self-host 운영자입니다 — NAS, 소형 VPS, 홈랩. 권한 계층은 하나뿐이며 멀티테넌트 모델은 없습니다.
</Callout>

## 1. 서버 실행 {#run-server}

Docker Compose로 서버를 부팅합니다. Docker 24+와 Compose v2가 필요합니다.

```bash title="terminal"
git clone https://github.com/idenn207/comax-secrets.git
cd comax-secrets

docker compose -f deploy/compose/docker-compose.yml up -d --build
# ✓ secret-server listening on :8080
# ✓ dashboard served at /
```

첫 부팅은 SQLite DB(`data/secrets.db`)와 AES-256 마스터 키(`keys/master.key`, mode `0600`)를 생성하고, **bootstrap admin 토큰을 로그에 단 한 번** 출력합니다. 지금 캡처하세요 — 다시 조회할 수 없습니다.

```bash title="terminal"
docker compose -f deploy/compose/docker-compose.yml logs secret-server \
  | grep -A 1 "bootstrap admin token"
# Comax Secrets: bootstrap admin token (shown ONCE):
#     comax_a1b2c3d4e5f6g7h8…
```

## 2. CLI 설치 {#install-cli}

소스에서 빌드합니다. `make build`는 `./bin/secret`(CLI)와 `./bin/secret-server`를 만듭니다. 리눅스 배포용 정적 바이너리는 `make xbuild`(amd64·arm64·armv7, `CGO_ENABLED=0`)로 교차 빌드합니다.

```bash title="terminal"
make build
export PATH="$PWD/bin:$PATH"

secret --help
```

## 3. 로그인 {#login}

1단계에서 캡처한 bearer 토큰으로 인증합니다. 평문 토큰은 `~/.config/comax/credentials.json`(mode `0600`)에만 저장되고, 서버는 SHA-256 해시만 보관합니다.

```bash title="terminal"
secret login --server http://localhost:8080 --token <bootstrap-token>
# ✓ Authenticated · credentials written to ~/.config/comax/credentials.json
```

토큰을 커맨드라인에 남기고 싶지 않다면 `$COMAX_TOKEN`으로 전달할 수 있습니다.

```bash title="terminal"
export COMAX_TOKEN=<bootstrap-token>
secret login --server http://localhost:8080
```

브라우저에서 `http://localhost:8080/`을 열고 같은 토큰을 붙여 넣어도 됩니다 — 대시보드는 `HttpOnly`·`Secure`·`SameSite=Strict` 세션 쿠키를 발급합니다.

## 4. 프로젝트 초기화 & import {#init-and-import}

관리할 디렉터리에서 프로젝트와 환경을 만듭니다. `secret init`은 서버에 프로젝트 + 환경을 생성하고 현재 디렉터리에 `.secretrc`(gitignore 됨)를 씁니다. 기존 `.env`가 있다면 `secret push`로 한 번에 올립니다 — 각 키는 대상 환경에 새 버전으로 업로드됩니다.

```bash title="terminal"
cd my-app
secret init --project checkout-api \
            --envs local,staging,prod --default-env local
# ✓ Created project checkout-api with 3 envs
# ✓ Wrote .secretrc

secret push --file .env
# ✓ Pushed 16 secrets to checkout-api/local
```

`secret init`은 멱등입니다 — 프로젝트나 환경이 이미 있으면 건너뜁니다.

## 5. 시크릿 주입 {#inject-secrets}

핵심 명령입니다. `secret run -- <cmd>`는 환경을 메모리로 pull해서 자식 프로세스에 환경변수로 주입합니다. **이 단계에서 어떤 것도 디스크에 쓰이지 않습니다.** 자식의 exit code가 그대로 전달되므로 CI에서 투명하게 동작합니다.

```bash title="terminal"
secret run -- node server.js
# ✓ Injected 16 secrets into the environment

secret run --env prod -- ./deploy.sh
```

파일이 필요하면 `secret pull`로 내보낼 수 있습니다. 상속과 참조가 해소된 최종 값이 기록됩니다.

```bash title="terminal"
secret pull                 # → .env (기본)
secret pull --out -         # → stdout
secret pull --env prod --out prod.env
```

## 명령어 레퍼런스 {#command-reference}

모든 명령은 `.secretrc`에 고정된 프로젝트·기본 환경을 따릅니다. `--env`로 대상을 덮어쓸 수 있습니다. 전체 플래그와 관리자 전용 명령(`token`, `webhook`)은 [CLI 레퍼런스](/docs/cli)를 참고하세요.

| 명령 | 동작 |
|---|---|
| `secret login` | bearer 토큰으로 인증하고 credentials를 기록합니다. |
| `secret init` | 프로젝트와 환경을 생성하고 `.secretrc`를 작성합니다. |
| `secret push --file .env` | 로컬 `.env`의 모든 키를 새 버전으로 업로드합니다. |
| `secret pull` | 상속·참조가 해소된 환경을 `.env`로 내보냅니다. |
| `secret run -- <cmd>` | 시크릿을 메모리로 주입하고 명령을 실행합니다. 자식 exit code를 전달합니다. |
| `secret set KEY=value` | 단일 키를 설정합니다. 새 버전이 생성됩니다. |
| `secret get KEY` | 단일 키의 값을 출력합니다 (뒤에 줄바꿈 없음). |
| `secret diff --against prod` | 현재 환경과 다른 환경 사이의 키 차이를 보여줍니다. |
| `secret export --format github-env` | 환경을 여러 형식(`dotenv`·`github-env`)으로 내보냅니다. |

## Config 상속 {#config-inheritance}

각 환경은 부모 환경을 `inherits_from`으로 지정할 수 있습니다. 이를테면 `prod`가 `staging`을, `staging`이 `local`을 상속합니다. 하위 환경이 같은 키를 정의하면 그 값이 우선하고(override), 정의하지 않은 키는 부모에서 상속받아 보입니다.

- 상속된 키는 대시보드에서 <Badge tone="neutral">상속</Badge>로 표시됩니다.
- 상속은 **pull 시점**에 해소되므로, 부모 값이 바뀌면 하위 환경에 자동 반영됩니다.
- 상속 체인의 순환(`dev → shared → dev`)은 cycle 감지로 거부되며, 서버는 순환 경로를 담은 `400`을 반환합니다.

## Secret 참조 {#secret-references}

한 값 안에서 다른 config의 값을 `${{ env.KEY }}` 형태로 참조합니다. 참조는 그냥 시크릿 값이므로 CLI로 바로 쓸 수 있습니다.

```bash title="terminal"
secret set DATABASE_URL='postgres://app:${{ prod.DB_PASSWORD }}@db/app' --env prod
```

- 참조 키는 대시보드에서 <Badge tone="info">ref</Badge>로 표시됩니다.
- 참조는 write 시점이 아니라 **pull 시점**에 해소됩니다 — 원본이 바뀌면 자동으로 반영됩니다.
- 참조된 환경 자체도 상속·중첩 참조까지 해소되므로 체인이 동작합니다.
- `A.K1 → ${{ B.K2 }} → ${{ A.K1 }}` 같은 순환은 parse error로 거부됩니다(cycle detection).

<Callout type="info">
`${{ }}` 밖의 `{{ }}`는 그대로 둡니다 — 이름 문자셋(`A-Za-z0-9_.+-`)을 벗어나면 리터럴로 취급하므로 JSON 템플릿 안에서 자유롭게 쓸 수 있습니다.
</Callout>

## 버전과 롤백 {#versioning-and-rollback}

모든 시크릿은 단조 증가하는 version과 append-only `secret_versions` 이력을 가집니다. 대시보드는 CLI가 잘 못하는 세 가지를 담당합니다: 버전 단위 diff, 단일 시크릿 롤백, 환경 간 diff.

- 롤백은 이전 값을 **새 버전으로** 다시 기록합니다 — 이력은 사라지지 않습니다.
- 모든 상태 변경은 `audit_log`에 기록됩니다 (acting token ID + 불투명 target).

<Callout type="warning">
롤백은 비가역 동작이지만, 롤백 후에도 이전 버전은 이력에 남습니다.
</Callout>

## 위협 모델 & 운영자 의무 {#threat-model}

self-host 모델은 운영자가 호스트·데이터베이스·마스터 키를 소유한다고 가정합니다. Comax는 managed 서비스가 아니며, 신뢰 경계는 그에 따라 그어집니다. 전체 목록은 [보안 모델](/docs/security)에 있습니다.

### 보호하는 것 {#what-it-protects}

- DB 디스크 이미지 유출 — 모든 값은 AES-256-GCM으로 암호화되어 키 없이는 불투명합니다.
- 토큰 유출 — 서버는 SHA-256 해시만 저장하며 평문은 발급 후 로깅하지 않습니다.
- `secret run` 중 시크릿이 디스크에 닿는 것 — 메모리에서 자식에 주입하며, 테스트가 새 파일 미생성을 검증합니다.

### 보호하지 않는 것 {#what-it-does-not-protect}

- 마스터 키 자체의 유출 — 키는 전체 신뢰 앵커입니다.
- 서버 호스트의 root 탈취, 개발 워크스테이션 침해.

### 운영자 의무 {#operator-duties}

- `keys/master.key`는 반드시 mode `0600` — Unix에서 더 넓으면 서버가 부팅을 거부합니다.
- `data/secrets.db`와 `keys/master.key`를 함께 백업하세요 — 서로 없으면 무용합니다.
- 이 바이너리는 평문 HTTP를 씁니다. localhost를 벗어나면 reverse proxy로 TLS를 종단하세요.
