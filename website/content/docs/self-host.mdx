Comax Secrets는 단일 바이너리로 설계되었으므로, `docker-compose.yml` 하나로 프로덕션급 배포를 구성할 수 있습니다.

## 아키텍처 {#architecture}

```
[외부 리버스 프록시 (Caddy/nginx/Traefik)]
    ↓ HTTPS
[host:8080 ← docker-compose 포트 바인딩]
    ↓
[secret-server 컨테이너]
    ├── /app/data/secrets.db (bind-mounted ./data/)
    ├── /app/keys/master.key (bind-mounted ./keys/)
    └── distroless/static (2 MB, 쉘 없음)
```

## Docker Compose 설정 {#docker-compose-setup}

`deploy/compose/docker-compose.yml` 사용:

```yaml
services:
  secret-server:
    image: comax-secrets-server:latest
    build:
      context: ../..
      dockerfile: deploy/docker/Dockerfile
    restart: unless-stopped
    ports:
      - "${COMAX_PORT:-8080}:8080"
    environment:
      COMAX_LISTEN: ":8080"
      COMAX_DB_PATH: "/app/data/secrets.db"
      COMAX_MASTER_KEY_FILE: "/app/keys/master.key"
      COMAX_AUTO_GENERATE_KEY: "1"
    volumes:
      - ./data:/app/data
      - ./keys:/app/keys
    healthcheck:
      test: ["CMD", "/app/secret-server", "--health", "--health-url", "http://127.0.0.1:8080/healthz"]
      interval: 10s
      timeout: 3s
      retries: 5
```

핵심:
- **bind-mount**: 호스트 `./data` 및 `./keys` 디렉터리를 컨테이너에 노출. 운영자가 파일 모드·소유권을 확인할 수 있음.
- **distroless**: 최종 이미지는 쉘·패키지 관리자 없는 `gcr.io/distroless/static:nonroot` (~2 MB). 공격 표면 최소.
- **자동 키 생성**: `COMAX_AUTO_GENERATE_KEY=1`은 첫 부팅 시 마스터 키를 생성 (이미 있으면 건너뜀).
- **healthcheck**: distroless에는 `curl`이 없으므로, 바이너리 자체가 `--health` 플래그로 헬스체크 역할.

## 첫 부팅 {#first-boot}

```bash
docker compose -f deploy/compose/docker-compose.yml up -d --build

# 부트스트랩 토큰 캡처 (한 번만 출력)
docker compose -f deploy/compose/docker-compose.yml logs secret-server \
  | grep -A 1 "bootstrap admin token"
```

## 마스터 키 관리 {#master-key-management}

마스터 키는 모든 저장된 시크릿을 암호화하는 신뢰 앵커입니다. 운영자 의무:

1. **파일 권한**: `./keys/master.key`는 모드 `0600`, 서버 사용자 소유. Unix에서는 서버가 부팅을 거부함.
2. **백업**: `./data/secrets.db`와 `./keys/master.key` **모두** 같은 위치·빈도로 백업. 둘 없이는 쓸모 없음.
3. **회전 금지 (M1)**: 키 회전은 모든 행을 다시 암호화하므로 M1에서는 미지원. 프로덕션에 가기 전에 계획하세요.

```bash
# 호스트에서 파일 권한 확인
ls -la ./keys/master.key
# -rw------- 1 user user 32 Jul 04 12:00 ./keys/master.key
```

## TLS 종료 {#tls-termination}

이 바이너리는 평문 HTTP를 사용합니다. 프로덕션 배포는 **반드시** 리버스 프록시를 앞에 두세요. 토큰은 bearer 자격증명이므로 TLS를 거쳐야 합니다.

### Caddy 예제 {#caddy-example}

```
secrets.example.com {
  reverse_proxy localhost:8080
}
```

### nginx 예제 {#nginx-example}

```nginx
server {
  listen 443 ssl http2;
  server_name secrets.example.com;
  ssl_certificate /etc/ssl/certs/cert.pem;
  ssl_certificate_key /etc/ssl/private/key.pem;

  location / {
    proxy_pass http://localhost:8080;
    proxy_set_header Host $host;
    proxy_set_header X-Real-IP $remote_addr;
    proxy_set_header X-Forwarded-Proto $scheme;
  }
}
```

## 데이터 보존 {#data-retention}

bind-mount를 사용하므로 `./data/` 및 `./keys/` 디렉터리는 호스트의 로컬 파일시스템에 있습니다. 백업 정책:

- 정기적 스냅샷: NAS 스냅샷, VM 백업, `rsync` 크론 작업
- 이중화: **마스터 키와 데이터베이스를 같은 빈도·위치에서 백업**
- 테스트: 복구 프로세스를 정기적으로 검증

## 외부 데이터베이스 {#external-database}

M1은 SQLite만 지원합니다. Postgres/Redis 지원은 나중 마일스톤 예정입니다.

## 환경 변수 {#environment-variables}

| 변수 | 기본값 | 설명 |
|---|---|---|
| `COMAX_LISTEN` | `:8080` | 바인드 주소:포트 |
| `COMAX_DB_PATH` | `./data/secrets.db` | SQLite DB 경로 |
| `COMAX_MASTER_KEY_FILE` | `./keys/master.key` | 마스터 키 파일 경로 |
| `COMAX_AUTO_GENERATE_KEY` | `1` | 첫 부팅 시 키 자동 생성 |
| `COMAX_WEBHOOK_POLL` | `10s` | 웹훅 배달 워커 폴링 주기 |
| `COMAX_WEBHOOK_ALLOW` | (없음) | 웹훅 URL CIDR 화이트리스트 |
| `COMAX_WEBHOOK_DENY` | (없음) | 웹훅 URL CIDR 블랙리스트 |

## 모니터링 {#monitoring}

healthcheck는 10초마다 `/healthz`를 확인합니다. 실패하면 컨테이너 상태가 `unhealthy`가 됩니다:

```bash
docker compose ps
# NAME           STATUS
# secret-server  Up 2 hours (healthy)
```

로그 모니터링:

```bash
docker compose logs -f secret-server
```
